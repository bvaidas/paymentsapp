Introduction

It’s a classic application exposing very basic Payments RESTful API. It contains minimal business logic to imitate transactions processing.

Please do not use this application in production.

Application Components

Application follows classic design pattern where data is stored one relational database and application runs in a container.

Application utilizes Spring Boot and following framework components

Component

Purpose

Spring Boot

Application Core framework.

Spring Boot Data JPA

Data access layer component. Used to manage application data. Might be replaced in the future with alternative datastore.

Spring Boot Web

Rest API component used to develop API controller.

Spring Rest Docs

Rest API documentation component. Used to generate API usage examples from testing data.

Relational database

Used to store application data. H2 database is used for testing purposes.

JDK 12 Container

Application runtime environment.

Docker

Container Engine used build Docker images.

Kubernetes/Openshift

Container platform used to run application. Application can run on pretty much any other container platform e.g. AWS ECS.

Building and Running the Application

Clone application source and build Docker image by running following command:

$ docker build -t payments-app .

You can test image locally by executing

$ docker run -p 8080:8080 payments-app

Promote image into external image repository and then deploy it on Kubernetes Cluster

$ kubectl run payments-app --image=<image_tag> --port=8080

Application by default application is using in-memory database. To make it look more like "production" deployment it should be linked with relational database container.

API usage

Further sections provides more details about each API endpoint usage. Samples are using http://localhost:8080 as an application URL.

Creating a Payment

You can create a Payment by performing POST request against API endpoint /v1/payments/.

Payload

Payment Request should be submitted as following payload

{
  "sender_name" : "Wilfred Jeremiah Owens",
  "sender_account_number" : "31926819",
  "sender_account_number_code" : "BBAN",
  "receiver_name" : "Emelia Jane Brown",
  "receiver_account_number" : "GB29XABC10161234567801",
  "receiver_account_number_code" : "IBAN",
  "payment_purpose" : "Server payment",
  "currency" : "GBP",
  "amount" : 200,
  "reference" : "Test reference"
}

Payload details

Path Type Description

sender_name

String

Sender name

sender_account_number

String

Sender account number

sender_account_number_code

String

Account sender name

receiver_name

String

Receiver name

receiver_account_number

String

Receiver Account Number

receiver_account_number_code

String

Receiver Account Number code

payment_purpose

String

Payment Purpose

amount

Number

Amount

currency

String

Currency code e.g. GBP

reference

String

Payment reference

Deleting Payment Details

You can delete Payment details by performing DELETE request against API endpoint /v1/payments/{id} where {id} is unique payment id.

Examples 

$ curl 'http://localhost:8080/v1/payments/987419da-5d19-44bd-bf14-a4756ce80af8/' -i -X DELETE \
    -H 'Accept: application/json'

API will respond will following response

HTTP/1.1 200 OK

List Payments

You can list existing payments by performing GET request against API endpoint /v1/payments/.

Successful operation will result with following response

$ curl 'http://localhost:8080/v1/payments/' -i -X GET \
    -H 'Accept: application/json'
